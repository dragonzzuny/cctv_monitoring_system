# P. CTO 리스크 보고서 (CTO Risk Report)

> 작성 기준: 30년 경력 시니어 아키텍트 — 경영진 대상 최종 보고서
> 대상 시스템: CCTV SAFE-YOLO Industrial Monitoring System
> 작성일: 2026-02-11
> 분류: 기밀 (Confidential)

---

## Executive Summary

**이 시스템은 "동작하는 프로토타입"이며, "프로덕션 소프트웨어"가 아니다.**

산업 안전 모니터링이라는 미션 크리티컬 도메인에서, 인증 없는 CCTV 접근, 에러 삼킴으로 인한 무탐지, 단일 장애점 아키텍처는 **법적, 안전, 운영 3중 리스크**를 내포한다.

---

## 리스크 매트릭스

```
영향도 ↑
        ┌──────────────┬──────────────┐
  치명  │ [R1] 보안 부재  │ [R2] 무탐지   │
        │ [R4] 단일장애점 │              │
        ├──────────────┼──────────────┤
  심각  │ [R3] 법적 리스크 │ [R5] 확장불가  │
        │ [R7] 테스트부재 │ [R6] 운영불가  │
        ├──────────────┼──────────────┤
  중간  │              │ [R8] 기술부채  │
        └──────────────┴──────────────┘
          높은 발생확률     중간 발생확률
```

## 리스크 상세

### R1. 보안 부재 — CRITICAL

| 항목 | 상세 |
|------|------|
| **현황** | 전체 시스템에 인증/인가 없음. CCTV 영상에 누구든 접근 가능. |
| **영향** | CCTV 영상 유출, 시스템 무단 조작, 안전 이벤트 은폐 가능 |
| **법적 리스크** | 개인정보보호법 위반 (영상정보 관리 의무), 산업안전보건법 위반 가능 |
| **발생 확률** | 네트워크에 노출 시 100% |
| **완화 비용** | 2주 (JWT + CORS + HTTPS) |
| **미조치 비용** | 법적 제재, 신뢰 상실, 사고 시 책임 확대 |

### R2. 탐지 실패의 침묵 — CRITICAL

| 항목 | 상세 |
|------|------|
| **현황** | 탐지기 에러 시 빈 결과 반환, 에러 로그만 남기고 계속 진행 |
| **영향** | 위험 상황(미착용 안전모, 위험구역 침입)이 탐지되지 않아도 시스템이 "정상"으로 표시 |
| **안전 리스크** | 작업자 안전사고 발생 시 시스템이 경고하지 못함 |
| **발생 확률** | GPU OOM, 모델 파일 손상, 네트워크 지연 시 발생 가능 — 중간~높음 |
| **완화 비용** | 1주 (에러 전파 + DEGRADED 상태 + 운영자 알림) |
| **미조치 비용** | 안전사고 발생 시 시스템 제조사 책임 가능 |

### R3. 법적 규정 미준수 — CRITICAL

| 항목 | 상세 |
|------|------|
| **현황** | 영상정보 접근 기록 없음, 보존 정책 없음, 암호화 없음 |
| **관련 법규** | 개인정보보호법 제25조(영상정보처리기기), 산업안전보건법 |
| **영향** | 행정 제재, 과태료, 형사 처벌 가능 |
| **완화 비용** | 1개월 (감사 로그 + 보존 정책 + 암호화 + 접근 제어) |

### R4. 단일 장애점 (SPOF) — HIGH

| 항목 | 상세 |
|------|------|
| **현황** | 단일 프로세스에 모든 기능 집중. 하나가 죽으면 전부 죽음 |
| **영향** | GPU OOM → API 서버 중단 → 모든 카메라 모니터링 중단 |
| **발생 확률** | 장기 운영 시 높음 (메모리 누적, GPU 문제) |
| **완화 비용** | 2~3개월 (프로세스 분리, 서킷 브레이커, 헬스체크) |

### R5. 확장 불가능 — HIGH

| 항목 | 상세 |
|------|------|
| **현황** | 카메라 2~3대 수준에서만 동작. SQLite 단일 쓰기 제한. |
| **영향** | 사업 확장 시 시스템 재작성 필요. 고객 확대 불가. |
| **발생 확률** | 사업 성장 시 100% |
| **완화 비용** | 3~6개월 (아키텍처 재설계) |

### R6. 운영 인프라 부재 — HIGH

| 항목 | 상세 |
|------|------|
| **현황** | 모니터링 없음, 알림 없음, CI/CD 없음, 백업 없음 |
| **영향** | 장애 인지까지 수시간~수일, 수동 배포로 인한 휴먼 에러 |
| **완화 비용** | 2~4주 (Docker + 모니터링 + CI/CD) |

### R7. 테스트 커버리지 0.5% — HIGH

| 항목 | 상세 |
|------|------|
| **현황** | 단 1개 테스트. 모든 변경이 회귀 테스트 없이 배포됨. |
| **영향** | 버그 수정이 새 버그를 도입. 리팩토링 불가능. |
| **완화 비용** | 2~3주 (핵심 모듈 테스트 60% 커버리지) |

### R8. 기술 부채 누적 — MEDIUM

| 항목 | 상세 |
|------|------|
| **현황** | 전역 싱글턴, God Module, 에러 삼킴, 하드코딩 전반 |
| **영향** | 기능 추가 비용이 기하급수적으로 증가. 개발 속도 저하. |
| **장기 영향** | 6개월 후 유지보수 불가능 상태 도달 예상 |
| **완화 비용** | 지속적 (리팩토링 로드맵 Phase 1~3) |

## 최종 판단

**이 시스템은 기술적으로 능숙한 개발자가 만든 "잘 동작하는 프로토타입"이다.**

도메인 지식, AI/ML 통합, 실시간 스트리밍 구현 — 핵심 기능은 잘 구현되어 있다. 문제는 이것이 프로토타입의 품질 수준에서 멈춰 있다는 것이다.

**프로토타입에서 프로덕션으로의 전환에는 기능 개발의 3~5배 노력이 필요하다.** 보안, 테스트, 운영, 확장성 — 이것들은 "추가 기능"이 아니라 "프로덕션의 전제 조건"이다.

**시나리오 B(단계적 개선)를 강력히 권고한다. Phase 1(보안 패치)은 즉시 시작해야 한다.**
